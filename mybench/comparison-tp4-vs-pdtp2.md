# TP4 vs PD TP2 性能对比报告

- **Model**: `qwen/qwen3.5-27b-fp8`
- **Hardware**: 4× NVIDIA RTX PRO 4000 Blackwell (PCIe, no NVLink)
- **TP4 config**: Single server, GPU 0-3, port 8000
- **PD TP2 config**: Prefill TP=2 (GPU 0-1, port 30000) + Decode TP=2 (GPU 2-3, port 30001) + Router (port 8000)
- **Transfer backend**: NIXL over PCIe
- **Test matrix**: 30 experiments (6 input lengths × 5 output lengths)
- **Date**: 2026-06-10

## 6. 大 Input + 大 Output 场景深度分析

### 现象观察

从吞吐对比数据可以观察到一个关键趋势：**当 input_len 和 output_len 都较大时，PD TP2 的吞吐逐渐接近 TP4**。

| 配置 | PD/TP4 吞吐比 | Decode 时间占比 |
|---|---:|---:|
| (128, 32) | 0.42× | 16% |
| (128, 512) | 0.65× | 79% |
| (2048, 32) | 0.73× | 8% |
| (2048, 512) | 0.89× | 77% |
| **(4096, 256)** | **0.88×** | **39%** |
| **(4096, 512)** | **0.93×** | **61%** |

### 数学解释

每个请求的总延迟可以分解为：

```
总延迟 = prefill_time + KV_transfer_time + decode_time
```

对于 PD 模式：
- `prefill_time ∝ input_len`（在 2 张 GPU 上并行计算）
- `KV_transfer_time ∝ input_len × KV_size / PCIe_bandwidth`（固定开销）
- `decode_time ∝ output_len × ITL`（逐 token 生成）

**关键点**：KV transfer 是一次性开销，而 decode 工作量与 output_len 成正比。

以 (4096, 512) 为例：
- TTFT（含 prefill + KV transfer）: 10.8s
- Decode 时间: 512 × 33ms = 16.9s
- **Decode 占比**: 16.9 / (10.8 + 16.9) = **61%**

当 output_len 足够大时，decode 工作占总延迟的 60-80%，KV transfer 的固定开销被摊薄，PD 的吞吐劣势从 60% 缩小到 7%。

### 现实意义

这意味着 **PD 分离在"长 input + 长 output"场景下具有接近 TP4 的吞吐潜力**，同时保留了 ITL 的显著优势（0.48×，快 2 倍）。

## 7. 现实场景匹配

以下场景符合"长 input + 长 output"特征，是 PD 分离的理想应用场景：

### 7.1 文档摘要与生成
- **Input**: 长文档（4K-8K tokens）
- **Output**: 详细摘要或报告（500-2000 tokens）
- **例**: 法律合同分析、研究论文总结、财报解读

### 7.2 代码生成与重构
- **Input**: 大型代码库上下文（4K+ tokens）
- **Output**: 完整函数或模块实现（500+ tokens）
- **例**: 根据需求生成完整功能、大规模重构、跨文件修改

### 7.3 长对话 Agent
- **Input**: 累积的多轮对话历史 + 工具调用结果（4K+ tokens）
- **Output**: 复杂的推理链或行动计划（500+ tokens）
- **例**: 自主编程 Agent、复杂任务规划、多步决策系统

### 7.4 长文档翻译
- **Input**: 源语言文档（4K+ tokens）
- **Output**: 目标语言翻译（类似长度，4K+ tokens）
- **例**: 技术文档、书籍翻译、法律文本

### 7.5 RAG 长检索 + 详细回答
- **Input**: 检索到的多个文档片段（4K-8K tokens）
- **Output**: 综合性详细回答（500-1000 tokens）
- **例**: 企业知识库问答、客服系统、技术支持

### 7.6 创意写作与内容生成
- **Input**: 详细的背景设定、参考资料（4K+ tokens）
- **Output**: 长篇文章、故事、营销文案（1000+ tokens）
- **例**: 小说创作、品牌故事、深度分析报告

**共同特征**：需要处理大量上下文，并生成实质性长度的输出。

## 8. KV Cache 通信-计算融合优化

当前 PD 模式的瓶颈在于 KV cache 传输是串行执行的：

```
当前流程（串行）：
[prefill 全部] → [KV transfer 全部] → [decode 开始]
时间：2-3s      + 8-10s              + 16.9s = ~27s
```

通过通信-计算融合优化，可以显著隐藏 KV 传输开销。

### 8.1 分块 Prefill + 流式 KV 传输（最高价值，P0）

**核心思想**：不等 prefill 全部完成再传 KV，而是边算边传。

```
优化后流程（流水线）：
[prefill chunk 1] → [transfer chunk 1 + prefill chunk 2] → [transfer chunk 2 + decode step 1] → ...
时间：0.5s         + 0.5s (overlap)                        + 16.9s = ~17.5s
```

**实现方式**：
- 将 input 分成 N 个 chunk（如每 chunk 512 tokens）
- Prefill 一个 chunk 后立即开始传输该 chunk 的 KV cache
- Decode worker 收到第一个 chunk 的 KV 后就可以开始生成（使用 partial attention）

**预期收益**：
- KV transfer 时间从 8-10s 降低到 2-3s（与 prefill 重叠）
- TTFT 从 10.8s 降低到 4-5s
- 吞吐提升 **20-30%**，(4096, 512) 配置下 PD/TP4 比率可达 **1.1×**（PD 反超）

**复杂度**：中等
- 需要修改 scheduler 支持 chunked prefill
- 需要 KV transfer 支持 streaming
- SGLang 已有 chunked prefill 基础，扩展即可

### 8.2 KV Cache 量化压缩（中等价值，P1）

**核心思想**：传输前压缩 KV cache，减少传输量。

```
当前：FP16 KV cache → 直接传输
优化：FP16 → INT8/FP8 量化 → 传输 → 解码端反量化
```

**实现方式**：
- Prefill 端：KV cache 从 FP16 量化到 INT8（或 FP8）
- 传输：数据量减少 2×（FP16→INT8）或 2×（FP16→FP8）
- Decode 端：反量化回 FP16 用于 attention

**预期收益**：
- KV transfer 时间减少 50%
- TTFT 从 10.8s 降低到 7-8s
- 吞吐提升 **10-15%**
- 可能引入轻微的精度损失（通常 <1%）

**复杂度**：低
- 标准量化技术，成熟方案
- 可在 NIXL 传输层集成

### 8.3 请求级流水线（中等价值，P1）

**核心思想**：当前请求的 KV 传输与下一个请求的 prefill 重叠。

```
Request 1: [prefill] → [KV transfer] → [decode]
Request 2:           [prefill] → [KV transfer] → [decode]
                    ↑ overlap ↑
```

**实现方式**：
- Prefill worker 维护一个请求队列
- 当前请求的 KV 传输时，同时开始下一个请求的 prefill
- 需要 GPU 支持并发计算（prefill + transfer DMA）

**预期收益**：
- 提高 prefill worker 利用率
- 在高并发场景下吞吐提升 **15-25%**
- 对单请求 TTFT 无改善

**复杂度**：中等
- 需要精细的调度和资源管理
- 避免 prefill 和 transfer 争抢 GPU 带宽

### 8.4 GPU Direct RDMA（中等价值，P2）

**核心思想**：绕过 CPU，直接 GPU-to-GPU 传输。

```
当前：GPU → CPU memory → NIC → CPU memory → GPU
优化：GPU → NIC → GPU (RDMA)
```

**实现方式**：
- 使用 GPUDirect RDMA 技术
- NIXL 已支持，但需要正确配置
- 需要网卡和驱动支持

**预期收益**：
- 减少 CPU 拷贝开销
- 延迟降低 **10-20%**
- 带宽利用率提升

**复杂度**：低-中
- 主要是配置问题
- 需要验证硬件兼容性

### 8.5 选择性 KV 传输（低价值，P3）

**核心思想**：只传输重要的 KV cache（基于 attention 重要性评分）。

```
当前：传输所有 KV cache
优化：只对 top-K% 重要的 KV 进行传输
```

**实现方式**：
- 在 prefill 阶段计算每个 KV 的 attention score
- 只传输 score 超过阈值的 KV
- Decode 端对缺失的 KV 使用近似或跳过

**预期收益**：
- KV 传输量减少 30-50%（取决于稀疏度）
- 可能引入精度损失（需要仔细调参）

**复杂度**：高
- 需要修改 attention 计算逻辑
- 精度影响难以预测

**结论**：价值较低，除非模型本身有高度稀疏的 attention pattern。

### 8.6 优化价值评估

| 技术 | 预期吞吐提升 | 实现复杂度 | 适用场景 | 优先级 |
|---|---:|---:|---|---:|
| 分块 Prefill + 流式 KV | **20-30%** | 中等 | 长 input + 长 output | **P0** |
| KV Cache 量化 | 10-15% | 低 | 所有场景 | **P1** |
| 请求级流水线 | 15-25% | 中等 | 高并发场景 | **P1** |
| GPU Direct RDMA | 10-20% | 低-中 | 有 RDMA 硬件 | **P2** |
| 选择性 KV 传输 | 10-20% | 高 | 稀疏 attention | P3 |

### 8.7 组合优化效果预估

如果同时应用 **分块 Prefill + KV 量化 + 请求级流水线**：

```
当前 (4096, 512)：
- TTFT: 10.8s
- Decode: 16.9s
- Total: 27.7s
- Throughput: 354 tok/s (0.93× TP4)

优化后：
- TTFT: 4-5s (分块 + 量化)
- Decode: 16.9s
- Total: 21-22s
- Throughput: 440-460 tok/s (1.15-1.2× TP4)
```

**结论**：PD 模式在大 input + 大 output 场景下**有反超 TP4 的潜力**，关键是通过通信-计算融合优化来隐藏 KV 传输开销。

## 9. 综合分析

### 胜者统计

| 指标 | TP4 胜 | PD TP2 胜 | 备注 |
|---|---|---|---|
| Output Throughput | ✅ 30/30 | ❌ 0/30 | TP4 全面碾压 |
| Mean TTFT | ✅ 30/30 | ❌ 0/30 | PD 受 PCIe KV 传输拖累 |
| Mean ITL | ❌ 5/30 | ✅ 25/30 | PD 在长 input 场景下优势巨大 |
| Mean TPOT | ❌ 5/30 | ✅ 25/30 | 同上 |
| P99 ITL | ❌ 0/30 | ✅ 30/30 | PD 长尾延迟控制极佳 |

### 关键发现

1. **吞吐瓶颈在 TTFT**
   - PD 模式下，每个请求都要等待 KV cache 从 prefill worker 传输到 decode worker
   - PCIe 带宽（~64 GB/s）远低于 GPU 内部显存带宽（~1 TB/s）
   - TTFT 占总延迟的 60-90%，导致 decode GPU 利用率低

2. **ITL 优势来自计算隔离**
   - TP4 模式下，prefill 和 decode 共享 GPU，prefill 计算会抢占 decode 资源
   - PD 模式下，decode worker 专注生成 token，不受 prefill 干扰
   - ITL 几乎恒定（20-35ms），不随 input length 增长

3. **长尾延迟（P99 ITL）差异巨大**
   - TP4 的 P99 ITL 在长 input 下会达到 3-6 秒（prefill 抢占导致）
   - PD 的 P99 ITL 稳定在 31-37ms，非常适合对延迟敏感的应用

4. **硬件限制是根本原因**
   - 你的硬件没有 NVLink，只有 PCIe 互联
   - KV cache 传输成为 PD 模式的致命瓶颈
   - 在有 NVLink（900 GB/s）的硬件上，PD 的吞吐劣势会大幅缩小

5. **大 input + 大 output 场景有优化潜力**
   - 当前 (4096, 512) 配置下 PD 已达 TP4 的 93%
   - 通过通信-计算融合优化，PD 可反超 TP4（预期 1.15-1.2×）
   - 这是 PD 分离的理想应用场景

---

## 10. 场景推荐

### 选择 TP4 的场景

- ✅ **追求最大吞吐**：所有配置下 TP4 吞吐都更高
- ✅ **短 input 场景**（input ≤ 512）：PD 的 TTFT 劣势更明显
- ✅ **高并发批量处理**：吞吐优先，延迟可接受一定波动
- ✅ **当前硬件条件**（无 NVLink）：TP4 是更稳妥的选择

### 选择 PD TP2 的场景

- ✅ **追求最低 ITL**：长 input 场景下 ITL 可快 10-33 倍
- ✅ **对延迟敏感的应用**：实时对话、流式生成、交互式 AI
- ✅ **长尾延迟控制**：P99 ITL 稳定在 35ms 以内
- ✅ **未来升级到 NVLink 硬件**：PD 的吞吐劣势会大幅改善

### 混合策略

如果业务场景混合（既有高吞吐需求，又有低延迟需求），可以考虑：

1. **双集群部署**：TP4 集群处理批量任务，PD 集群处理实时交互
2. **动态路由**：根据请求特征（input length、SLA 要求）路由到不同集群
3. **等待硬件升级**：升级到 NVLink 硬件后，PD 的吞吐劣势会缩小到 0.85-0.95×

## 11. 行动计划

基于以上分析，我们制定以下分阶段的行动计划：

### 11.1 短期行动（1-2 周）

1. **扩展测试矩阵**

2. **添加细粒度监控指标**

   - 确认是否是 PCIe 带宽瓶颈，还是 CPU 拷贝开销

   ```bash
   # 快速验证 FP8 KV cache 的精度影响

   ```
   - 对比 FP16 vs FP8 的模型输出质量
   - 测量 FP8 下的吞吐提升

### 11.2 中期行动（1-3 个月）

**目标**：实现核心优化，PD 在大 input + 大 output 场景下达到 TP4 性能

1. **实现分块 Prefill + 流式 KV 传输（P0）**
   ```python
   # 伪代码示例
   class ChunkedPrefillScheduler:
       def process_request(self, req):
           chunks = self.split_into_chunks(req.input_ids, chunk_size=512)
           for i, chunk in enumerate(chunks):
               kv_cache = self.prefill(chunk)
               # 异步传输当前 chunk 的 KV cache
               self.transfer_kv_async(kv_cache, decode_worker)
               if i == 0:
                   # 第一个 chunk 传输完成后，decode worker 可以开始生成
                   self.signal_decode_start(req)
   ```
   
   **关键设计点**：
   - Chunk size 选择：512-1024 tokens（平衡 prefill 效率和传输延迟）
   - Decode worker 需要支持 partial attention（只有部分 KV cache）
   - 需要处理 chunk 传输失败的重试机制
   
   **预期收益**：
   - (4096, 512) 配置下吞吐从 354 tok/s 提升到 440-460 tok/s
   - TTFT 从 10.8s 降低到 4-5s

2. **实现请求级流水线（P1）**
   ```python
   # Prefill worker 调度逻辑
   class PipelinedPrefillWorker:
       def run(self):
           while True:
               req = self.queue.get()
               # 异步启动 KV 传输
               transfer_future = self.transfer_kv_async(req)
               # 同时开始下一个请求的 prefill
               next_req = self.queue.get()
               kv_cache = self.prefill(next_req)
               # 等待上一个传输完成
               transfer_future.wait()
   ```
   
   **预期收益**：
   - 高并发场景下吞吐提升 15-25%
   - Prefill GPU 利用率从 60-70% 提升到 85-90%

3. **优化 Decode Worker 调度**
   - 实现 KV cache 预取（prefetch）机制
   - 支持多个请求的 KV cache 并行传输
   - 优化 attention kernel 以支持 partial KV cache

4. **建立 A/B 测试框架**
   ```python
   # 对比优化前后的性能
   class ABTest:
       def run(self, config_a, config_b, workload):
           results_a = self.benchmark(config_a, workload)
           results_b = self.benchmark(config_b, workload)
           return self.compare(results_a, results_b)
   ```

### 11.3 长期行动（3-6 个月）

**目标**：硬件升级，PD 成为默认部署模式

1. **评估 NVLink 硬件升级**
   - 对比当前 PCIe 3.0 x16（~32 GB/s）vs NVLink 4.0（~900 GB/s）
   - 预期：KV 传输时间从 8-10s 降低到 0.3-0.5s
   - 预期：PD 在所有场景下达到 TP4 的 95-105% 性能
   
   **成本效益分析**：
   ```
   当前配置（PCIe）：
   - 4x RTX 4000 Blackwell
   - 总成本：~$8,000
   - PD 性能：0.4-0.9× TP4
   
   升级配置（NVLink）：
   - 4x A100 80GB + NVSwitch
   - 总成本：~$40,000
   - PD 性能：0.95-1.05× TP4
   
   ROI：如果业务需要低延迟（ITL），NVLink 升级可以在 6-12 个月内收回成本
   ```

2. **探索更激进的优化**
   - **预测性 KV 传输**：基于历史请求模式，预测下一个请求的 KV cache 需求
   - **动态 KV 压缩**：根据 attention pattern 自适应调整 KV 量化精度
   - **与 Speculative Decoding 结合**：PD + 投机解码，进一步提升吞吐

3. **建立自动化调优系统**
   ```python
   class AutoTuner:
       def tune(self, workload_profile):
           # 根据工作负载自动选择最优配置
           if workload_profile.avg_input_len > 4000 and workload_profile.avg_output_len > 500:
               return "PD-TP2 with chunked prefill"
           elif workload_profile.p99_itl_sla < 50:
               return "PD-TP2"
           else:
               return "TP4"
   ```

### 11.4 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
